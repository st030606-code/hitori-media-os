# API なしワークフロー

MVP は OpenAI API、Anthropic API、その他の有料 AI 連携なしで動くようにします。

これは妥協ではなく、意図的な設計です。最初のバージョンでは、自動化より先にクリエイターの実作業を検証します。

## 使うツール

このワークフローでは、次のものを使えます。

- Codex
- Claude Code
- ChatGPT
- Claude アプリ
- Sanity
- ローカル JSON ファイル
- ローカル Markdown ファイル
- 保存済みプロンプト

## 外部メディア生成サービスの扱い

MVP では、OpenAI API や Anthropic API のような LLM API 連携は追加しません。

ただし、音声・画像・動画などのメディア制作については、既に契約している外部サービスを必要に応じて使えます。

例:

- ElevenLabs
- Fish Audio
- ChatGPT 画像生成
- CapCut
- その他、制作工程で必要なツール

これらは、コンテンツ生成の中核を API 自動化するものではなく、媒体化のための制作ツールとして扱います。

## フォルダの役割

- `inputs/content-ideas/`: Sanity から書き出した、またはローカルで作成した元レコード
- `prompts/`: 再利用する生成プロンプト
- `outputs/`: 生成・編集したプラットフォーム別の成果物
- `docs/devlog/`: 判断、発見、プロジェクト履歴

## 手動生成フロー

1. `inputs/content-ideas/` から入力レコードをひとつ選ぶ。
2. `prompts/` から目的に合うプロンプトをひとつ選ぶ。
3. 入力レコードとプロンプトを使いたい AI ツールに貼り付ける。
4. 必要な出力形式だけを生成するよう依頼する。
5. 下書きを対応する `outputs/` ディレクトリに保存する。
6. 人間が正確性、声色、公開可能性を確認する。
7. 何度も修正が必要だった点はプロンプトへ反映する。
8. ワークフロー上の重要な学びは devlog に残す。

## ファイル命名案

生成下書きには、次の形式を使います。

```text
YYYY-MM-DD--source-slug--output-type.md
```

例:

```text
2026-05-11--ai-blog-database--note.md
```

## 品質チェック

公開前に、次を確認します。

- 元レコードの主張を守っているか
- 元レコードにない主張を勝手に追加していないか
- 対象プラットフォームに合っているか
- 発信者本人の声に近いか
- 終わり方や次の行動が明確か
- 出典や注意点が誠実に扱われているか

## なぜ API から始めないのか

API は、ワークフローが安定してから役に立ちます。

最初から API 化すると、もっと重要な問いが見えにくくなります。

- コンテンツレコードには何を入れるべきか
- どの出力フォーマットが本当に役に立つか
- どのプロンプト変数が効くか
- どの工程に人間の判断が必要か
- どこを自動化すると、逆に手直しが増えるか

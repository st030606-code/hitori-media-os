# ロードマップ

このロードマップでは、コードによる大きな自動化より先に、手動ワークフローの検証を優先します。

## Phase 0: リポジトリの基礎づくり

状態: 開始済み

- ドキュメント構造を作る。
- プロンプトテンプレートを作る。
- ローカル入力・出力フォルダを作る。
- サンプルのコンテンツレコードを追加する。
- devlog 構造を作る。

## Phase 1: 手動コンテンツワークフロー

目的: ひとつの構造化レコードから、複数プラットフォーム向けの有用な下書きを作れるか検証する。

- `inputs/content-ideas/example-ai-blog-db.json` を改善する。
- note、Substack、Threads、X、YouTube、Shorts、Podcast、Diagram のサンプル出力を作る。
- 人間が確認した出力例を追加する。
- 生成品質を見ながらプロンプトを改善する。
- レコードと出力ファイルの命名ルールを決める。

## Phase 1.5: 出力レビュー

目的: 生成された下書きが、実際に公開できる品質に近いか確認する。

この工程を挟むことで、Sanity スキーマが机上の設計だけで固まることを防ぎます。

- 各出力に対して、人間の編集量を記録する。
- 使えたプロンプト変数と不要だった変数を整理する。
- note、Substack、Threads、X、YouTube で必要な情報の違いを確認する。
- スキーマに入れるべき項目と、プロンプト側で扱うべき項目を分ける。
- 公開に近い品質まで持っていけた出力例を `examples/` または `outputs/` に残す。

## Phase 2: スキーマ設計

目的: 再利用できる創作知識を保存するための Sanity スキーマを設計する。

- `contentIdea` ドキュメントを定義する。
- 主張、根拠、具体例、読者、反論、出力計画などの補助オブジェクトを定義する。
- MVP で必須にする項目を決める。
- 編集状態とレビュー手順を整理する。
- スキーマ項目とプロンプト変数を対応させる。

## Phase 3: Sanity 初期化

目的: 手動で検証したスキーマをもとに Sanity を初期化する。

- Sanity プロジェクトを作る。
- コンテンツスキーマを追加する。
- サンプルレコードを登録する。
- ひとつのレコードをローカル JSON に書き出す。
- 書き出したデータで既存プロンプトが使えるか確認する。

## Phase 4: 最小限のクリエイターダッシュボード

目的: コンテンツモデルが安定してから、小さな画面を作る。

- Next.js アプリを追加する。
- コンテンツレコードを表示する。
- 出力の準備状況を見られるようにする。
- ローカルプロンプトワークフローへの導線を作る。
- 投稿自動化を作り込みすぎない。

## Phase 5: 必要な部分だけ自動化

目的: 手動で有効性が確認できた作業だけを自動化する。

- ローカルでプロンプトを組み立てる補助スクリプトを追加する。
- 出力ファイル作成の補助を追加する。
- 繰り返し作業を減らせる場合だけ API 連携を検討する。
- 手動で戻れるワークフローは必ず残す。

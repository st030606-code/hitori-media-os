# 0014: ダッシュボードファーストのプロダクト構成を明確化

日付: 2026-05-12

## 背景

Sanity Studioの日本語ラベルを整備したことで、Studio上で初回入力を試せる状態になりました。

ただし、Sanity Studioを買い手の日常操作UIとして最適化しすぎると、このプロダクトの本来の価値からズレる可能性があります。

Sanity AI Content OS の長期的な主UIは、Sanity Studioではなく、将来のNext.jsダッシュボードです。

## 決定・変更

`docs/05-future-dashboard.md` を追加しました。

このドキュメントでは、次を整理しています。

- Sanityはデータバックエンドであること
- Sanity Studioは開発者・管理者・上級ユーザー向けの管理画面であること
- 買い手向けの日常操作UIは将来のNext.jsダッシュボードにすること
- 生アイデア入力、AI構造化、人間レビュー、出力生成、パイプライン追跡、公開後フィードバックの流れ

`docs/04-first-content-entry.md` には、Studioでの手入力は初期検証・管理者テストであり、長期的にユーザーへ全フィールド入力を求めるものではないと追記しました。

`docs/02-schema-design.md` には、スキーマ設計がSanity Studioと将来ダッシュボードの両方を支える必要があることを追記しました。

また、将来フィールドとして `contentIdea.rawInput` を追加する方針と、`platform` 値を自由入力ではなく制御値にする方針を明記しました。

## no-API MVPの維持

今回の変更はプロダクト設計ドキュメントの更新のみです。

Next.js、有料LLM API連携、OpenAI API / Anthropic API クライアント、自動生成、自動投稿は追加していません。

実project ID、APIキー、トークン、認証情報、シークレットも追加していません。

## 重要な判断

Sanity Studioは重要ですが、買い手向けの主要操作画面ではありません。

Studioはバックオフィス、デバッグ、初期検証、高度な手動編集に使います。

将来のNext.jsダッシュボードでは、ユーザーが最小限の生アイデアを入力し、AI支援で構造化し、人間が確認してからSanityへ保存する流れを目指します。

## 次に確認すること

- `contentIdea.rawInput` をTypeScriptスキーマに追加するか
- `platform` 系フィールドをselect optionsに統一するか
- 初回 `contentIdea` 入力ガイドに、ダッシュボード化で隠すべき入力項目を追記するか
- Next.jsを初期化する前に、ダッシュボードの画面設計をもう少し文書化するか

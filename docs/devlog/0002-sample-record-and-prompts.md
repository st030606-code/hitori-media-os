# 0002: サンプルレコードとプロンプト改善

日付: 2026-05-11

## 背景

Phase 1 の手動コンテンツワークフローへ進む前に、`inputs/content-ideas/example-ai-blog-db.json` を実際に複数出力へ展開できる強いサンプルレコードにする必要がありました。

テーマは「これからのブログは、記事を書くより『AIが使えるDB』を作った人が勝つ」です。

## 決定・変更

サンプルレコードを、単なる説明用 JSON ではなく、note、Substack、Threads、Instagram、YouTube、Shorts、Podcast、GitHub ドキュメントへ展開できる元レコードとして改善しました。

また、`prompts/` 配下の各プロンプトを、入力レコードのフィールドを明示的に使う形へ更新しました。

主に強化した項目:

- `coreThesis`
- `audiencePain`
- `claims`
- `evidence`
- `examples`
- `objections`
- `tone`
- `platformAngles`
- `outputChecklist`

## 理由

このプロジェクトでは、記事本文そのものではなく、複数媒体へ展開できる構造化された知識レコードを中心にします。

そのため、サンプルレコードには、AIが媒体別に変換しやすい情報を十分に入れる必要があります。

プロンプト側でも、単に「このJSONから作ってください」と言うだけでは弱いため、どのフィールドをどのように使うかを明示しました。

## APIを使わずに済ませた理由

今回の改善は、API連携なしで完結します。

Codex、Claude Code、ChatGPT、Claude アプリに、JSONレコードとプロンプトを貼り付けるだけで手動生成を試せます。

この段階では、自動生成パイプラインよりも、どの入力項目が出力品質に効くかを確認することが重要です。

## Codex と Claude Code の使い方

Codex は、リポジトリ構造、ドキュメント、プロンプト、devlog を整える実装・レビュー担当として使います。

Claude Code は、ローカルで入力レコードとプロンプトを扱い、生成物の下書き、編集、レビュー補助に使います。

どちらも、OpenAI API や Anthropic API を組み込むためではなく、人間が確認できる手動・半自動ワークフローを進めるために使います。

## no-APIワークフローへの貢献

今回の変更により、次の検証がしやすくなりました。

- ひとつの元レコードから複数出力を作れるか
- 各媒体で必要な情報の違いは何か
- プロンプトで参照すべきフィールドは何か
- 人間の編集量が多くなる箇所はどこか
- Sanity スキーマに昇格すべき項目は何か

## 次にテストすること

まだ実際の出力ファイルは作りません。

次のステップでは、このサンプルレコードと各プロンプトを使って、まず次の出力を手動生成し、Phase 1.5 の観点でレビューします。

- note 記事
- Substack 投稿
- Threads 投稿
- YouTube 長尺台本
- Shorts 台本
- Podcast 台本

レビューでは、生成物そのものの良し悪しだけでなく、どのフィールドが効いたか、どのフィールドが足りなかったかを記録します。

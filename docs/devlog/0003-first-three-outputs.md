# 0003: 最初の3出力を作成

日付: 2026-05-11

## 背景

Phase 1 の手動コンテンツワークフロー検証として、まず文章系の3媒体だけを試しました。

対象にした出力は note、Substack、Threads です。

YouTube、Shorts、Podcast、Diagram はまだ作成していません。

## 作成した出力

- `outputs/note/2026-05-11--ai-blog-db--note.md`
- `outputs/substack/2026-05-11--ai-blog-db--substack.md`
- `outputs/threads/2026-05-11--ai-blog-db--threads.md`

どれも公開用の完成稿ではなく、Phase 1.5 のレビューに回すための下書きです。

## 有効だったフィールド

特に効いたフィールド:

- `coreThesis`: 3媒体すべての中心軸として機能した。
- `audiencePain`: 導入や問題提起を作る材料になった。
- `claims`: noteの見出し、Threadsの投稿分解、Substack本文の骨格に使えた。
- `examples`: 抽象論を避けるために有効だった。
- `objections`: noteとSubstackで読者の不安に答えるセクションを作りやすかった。
- `platformAngles`: 媒体ごとの切り口の違いを出すのに役立った。
- `tone`: AIっぽい量産感を避けるための制約として使えた。

## プロンプトがうまく機能した点

noteプロンプトは、論考としての構造を作りやすかったです。

Substackプロンプトは、制作ログや個人的な気づきのトーンへ寄せやすかったです。

Threadsプロンプトは、ひとつの主張を短い投稿へ分解する用途に合っていました。

また、各プロンプトに保存先と人間による確認ポイントを入れておいたことで、出力後の扱いが分かりやすくなりました。

## note / Substack / Threads の比較

同じ元レコードから作っても、3媒体の役割はかなり違いました。

noteは、主張を深く説明する論考に向いています。`claims` と `objections` が特に効きました。

Substackは、制作過程や今週やったことを混ぜると、個人的なビルドログとして読ませやすくなります。`platformAngles` と devlog の情報が効きます。

Threadsは、`claims` を短く分解するのに向いています。ただし、投稿ごとの改行、長さ、1投稿1メッセージの制御が重要です。

## このテストで分かったこと

AIは、媒体ごとの最初の下書きを作る用途にはかなり使いやすいです。

一方で、そのまま最終公開できる文章になるとは限りません。

特に、発信者本人の体験、実際の作業画面、具体的な迷い、言葉の温度感は、人間が足したほうが強くなります。

つまり、AIの役割は「最終稿を一発で作ること」ではなく、「同じ元レコードから媒体別の初稿をすばやく作ること」と考えたほうがよさそうです。

## 人間レビューが必要そうな点

noteは、公開前に発信者本人の具体的な体験や実例を足したほうが強くなります。

Substackは、より個人的な制作ログとして読ませるなら、実際の作業画面、悩み、判断の迷いを追加したほうがよさそうです。

Threadsは、投稿ごとの長さや改行リズムを、実際の投稿画面に合わせて調整する必要があります。

3媒体すべてに共通して、SanityやObsidianの説明が読者に伝わるかは確認が必要です。

## YouTube / Shorts / Podcast / Diagram の前に改善すること

次の媒体へ進む前に、次を改善するとよさそうです。

- サンプルレコードに、発信者本人の具体的な作業エピソードを追加する。
- `platformAngles` に、媒体ごとの想定尺や完成イメージをもう少し足す。
- `evidence` に、実際の手動ワークフローから得た観察を追加する。
- Threadsの文字量や投稿数の上限をプロンプト側でもう少し制御する。
- noteとSubstackの違いを、さらに明確にする。

## note / Substack の発信切り口

note向け:

- 「これからのブログは、記事を書くよりAIが使えるDBを作った人が勝つ」
- 「ひとつの知識レコードから複数媒体へ展開する発信ワークフロー」
- 「Obsidianで考え、Sanityで構造化するAI時代のブログ設計」

Substack向け:

- 「今週、ブログを書く前にJSONを整えていた話」
- 「APIを使わずに、AIコンテンツOSを手動で試している理由」
- 「note、Substack、Threadsを同じ元レコードから作ってみた」

## 次にテストすること

次は、今回作った3つの下書きを人間がレビューします。

確認したいこと:

- 3媒体で十分に書き分けられているか
- 元レコードの `coreThesis` が保たれているか
- unsupported claims が混ざっていないか
- 人間の編集量はどれくらいか
- 次にYouTube、Shorts、Podcast、Diagramへ進めるだけの情報が元レコードにあるか

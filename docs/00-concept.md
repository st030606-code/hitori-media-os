# コンセプト

Sanity AI Content OS は、ひとりメディア運営者が「一度考えたアイデア」を複数の媒体へ展開するためのコンテンツ運用システムです。

出発点は、ひとつの構造化された知識レコードです。そのレコードには、中心アイデア、読者、主張、根拠、具体例、反論、トーン、参照リンク、公開意図などを入れます。

そこから、保存済みプロンプトとローカルワークフローを使い、各プラットフォーム向けの成果物を作ります。

## プロダクトの仮説

多くの発信ワークフローでは、各プラットフォームごとに別々の文章を書き直します。

このプロジェクトでは、元になるアイデアそのものを資産として扱います。

note 記事、ニュースレター、スレッド、動画台本などは、その資産から派生する出力です。

## 基本ループ

1. 構造化されたアイデアを作る。
2. 主張、読者、根拠を明確にする。
3. Sanity またはローカル入力ファイルに保存する。
4. 保存済みプロンプトで媒体別の下書きを作る。
5. 人間が確認・編集する。
6. 公開する。
7. 学びをスキーマとプロンプトへ戻す。

## 主な利用者

教育、分析、技術、意見発信などを複数プラットフォームで行う、ひとりメディア運営者です。

利用する可能性があるもの:

- Sanity: 構造化された知識ベース
- Codex: リポジトリとワークフロー整備
- Claude Code: ローカル生成と編集
- ChatGPT / Claude アプリ: 対話的な下書き作成
- ローカルファイル: 入力と出力の保存場所

## Obsidian と Sanity の役割

Obsidian は、未整理の思考、リサーチ、ラフメモ、開発ログ、日々の気づきを扱う場所として使います。

Sanity は、その中から再利用・展開する価値がある知識を、構造化して保存する場所です。

つまり、Obsidian は考える場所、Sanity は発信へ展開できる知識資産を管理する場所として分けます。

## 設計原則

- ひとつの元レコードから、複数の出力へ展開する。
- 繰り返し使える程度に構造化しつつ、創作の自由度は残す。
- 人間のレビューをプロダクトの一部として扱う。
- API 自動化は、手動ワークフローが機能してから追加する。
- プロンプトはバージョン管理すべき資産として扱う。
- スキーマは「公開するための項目」だけでなく「考えるための項目」を持つ。

## 想定する出力

同じ元アイデアから、次のような成果物を作ります。

- note 記事
- Substack ニュースレター
- Threads 投稿
- Instagram カルーセル案
- YouTube 長尺台本
- YouTube Shorts 台本
- ポッドキャスト台本
- 図解・画像案
- GitHub ドキュメント
- 有料記事の構成案

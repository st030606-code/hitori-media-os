## Main Body Draft

### 1. 発信を頑張るほど消耗する構造

ひとり運営の発信で苦しくなる地点は、だいたい決まっています。

- 媒体が増えるほど作業時間が伸びる
- 同じテーマを媒体ごとに書き直している
- AIに渡す素材が毎回違うので、出力もぶれる
- 反応に追われて、設計の時間が消える
- 完成度を上げようとして、公開頻度が落ちる

これは「努力が足りない」状態ではありません。1つのアイデアを複数媒体へ展開する仕組みがない状態で、媒体だけ増やしているから起きる構造的な詰まりです。

なので自分は、まずこの構造を変えることに集中しました。発信量の前に、制作の型の問題だと割り切ったわけです。

> 「ひとり運営は『量』ではなく『型』で詰む」  
> これが今、自分が一番強く感じている認識です。

### 2. 中心にあるのは「1つの構造化されたContent Idea」

仕組みの中心に置いているのは、`contentIdea` という名前のレコードです。SanityというCMSに、次の項目を分けて保存しています。

- 中心となる主張（coreThesis）
- 想定読者（audience）
- 読者の悩み（audiencePain）
- 主張リスト（claims）
- 具体例（examples）
- 反論と返答（objections）
- トーン（voice / styleNotes / avoid）
- 媒体ごとの切り口（X / note / Substack / YouTube / Shorts / Podcast / Instagram / GitHub / paid）

たとえば「AIで『ひとりメディア運営OS』を作っている裏側」というテーマも、この構造で1レコードに整理してあります。

ここまで揃えてから、次のステップで媒体別の下書きを派生させます。これがあるかないかで、AIに頼んでも、人間が書いても、出力の安定感がかなり違ってきます。

> 想定画像挿入: contentIdea を中心にして note / Substack / X / Threads / YouTube / Shorts / Podcast / Instagram に展開している関係図（H2「中心にあるのは『1つの構造化されたContent Idea』」直下に配置）

### 3. AIに記事を書かせる前に、AIが読めるDBを作る

最近、AIで記事を量産する話をよく見ます。それ自体は否定しません。実際、自分も下書きはAIと一緒に作っています。

ただ、いきなりAIに長文を書かせると、トーンや主張がぶれます。

理由はシンプルで、AIに渡している素材が散らかっているからです。プロンプトを毎回書き直すより、素材側の構造を直すほうが効きます。

#### プロンプトより先に素材を直す

- 主張は主張のフィールドに
- 反論は反論のフィールドに
- 媒体別の切り口は媒体別のフィールドに

ここを揃えてから、初めて「このContent Idea から Substack Post の下書きを作って」と頼めるようになります。

#### 出力が安定する理由

- AIが文脈を再構成しなくて済む
- 媒体ごとに足す情報が決まっている
- 自分の発信思想（coreThesis）から外れにくい

逆にいうと、AI記事生成だけ取り入れて素材整備を後回しにすると、毎回ぶれた出力を人間が手直しし続けることになります。これは仕組み化ではありません。

### 4. 自動化は最後、まず手動で回る型を作る

もう1つ、強く意識しているのは「自動化を急がない」ことです。

API投稿、画像生成API、Sanity direct write、ダッシュボードUI。どれも将来は入るかもしれませんが、いまは「まだ入れていません」。

理由は、手動で1キャンペーン回してみないと、自動化すべき作業が見えないからです。

#### 自動化を後回しにする実利

- 本当に繰り返している作業だけを特定できる
- 自動化したあとに「やっぱり要らなかった」を避けられる
- 手動の段階で、人間レビューの余白を残せる
- 出力の質を、自分の判断で評価できる

実際に手動で動かしてみると、消耗していたのは「下書き作成」ではなく「公開判断」や「Sanityへの反映」だったりします。優先順位はここから決めます。

> 想定画像挿入: 「手動 → 半自動 → 自動」の段階イメージ。最初に自動化するのは下書き生成ではなく、公開判断や反映作業のほうがコスパが高い、という図（H2「自動化は最後、まず手動で回る型を作る」末尾に配置）

### 5. text / visual / video / audio を1つのアイデアから捌く

`contentIdea` から派生させているアウトプットは、いまの段階で次のとおりです。

- text: note 記事 / Substack Post / Substack Notes / X / Threads / Newsletter
- visual: 図解 / アイキャッチ / Instagram カルーセル / X 図解ペア / YouTube サムネ
- video: YouTube long-form / Shorts / Instagram Reels
- audio: Podcast / 音声サマリー / ai-clone / TTS draft（手動レビュー前提）

これを束ねるのが、ローカルの「Publish Package Builder」というスクリプトです。`outputs/<platform>/` にある下書きと、Visual Register で登録した画像、Sanity 更新用の patch JSON を読み、媒体別の `publish-packages/<platform>/<slug>/` フォルダにチェックリスト付きでまとめます。

公開そのものは手動です。Builder は「公開準備セット」を作るだけで、SNSやSubstackへ自動で投稿することはありません。

### 6. いま手動に残している作業

正直なところ、自動化しているのはまだ多くありません。

- 画像生成は ChatGPT 手動
- 下書きの最終編集は手動
- Sanity Studio への反映は手動
- 各媒体への公開・投稿も手動
- 公開URLを元レコードに戻すのも手動

「これって全然OSじゃないのでは？」と思う人もいるかもしれません。でも、手動で回る型を1キャンペーン分用意した時点で、次に自動化すべき場所がはっきり見えてきます。

#### 次に自動化を検討する候補

- patch JSON の Sanity 反映補助
- placeholder draft の検出
- publish package の再生成補助
- Substack subscriber 数や反応の手動記録テンプレ
- 教材から抽象化メモへの変換テンプレ

派手な自動投稿よりも、こうした地味な補助が、ひとり運営の体感を大きく変える気がしています。

### 7. 真似できる最初の1ステップ

もしあなたが、自分の発信を1つだけ仕組み化したい場合、最初におすすめなのはこれです。

> 1つのテーマだけでよいので、「主張」「読者」「根拠」「反論」「媒体別の切り口」を分けたメモを作る。

JSONでもMarkdownでもよいです。ツールは何でも構いません。

このメモがあるだけで、AIに下書きを頼んだときも、自分で書くときも、媒体ごとに「同じテーマを書き直している感覚」がかなり減ります。

仕組み化は、最初の1テーマで十分です。

# これからのブログは、記事を書くより「AIが使えるDB」を作った人が勝つ

AI時代のブログは、完成記事の置き場から、複数の発信物を生み出す知識データベースへ変わっていく。

## 導入

ブログを書くこと自体は、これからも大事です。

ただ、これから差がつくのは「記事を何本書いたか」だけではないと思っています。

むしろ重要になるのは、AIが使える形で自分の考えを保存できているかどうかです。

多くの発信者は、同じアイデアを何度も書き直しています。note用に長く書き、Substack用に少し個人的に書き、Threads用に短く分解し、YouTube用に台本へ直す。

でも、そのたびに主張、根拠、具体例、読者、トーンを思い出しているなら、毎回かなりの文脈づくりが発生しています。

そこで必要になるのが、記事そのものではなく、AIが使える知識DBです。

## 本文

### 記事の山ではなく、AIが使える素材を作る

完成記事だけを保存していると、あとから別の媒体へ展開するときに、もう一度文脈を分解する必要があります。

たとえば、ある記事の中に「主張」「根拠」「具体例」「反論への回答」が混ざっているとします。人間が読むには自然でも、別の出力へ変換するときには扱いにくい。

一方で、元レコードの段階で次のように分かれていたらどうでしょう。

- 中心となる主張
- 想定読者
- 読者の悩み
- 主張ごとの根拠
- 具体例
- 反論と回答
- 媒体別の切り口

この形なら、AIに渡したときに「note向けには論考として」「Substack向けには制作ログとして」「Threads向けには短い連投として」と変換しやすくなります。

つまり、ブログを完成記事の倉庫として見るのではなく、AIが再利用できる素材置き場として設計するということです。

たとえば、最小のレコードはこれくらいで始められます。

```json
{
  "title": "これからのブログはAIが使えるDBになる",
  "coreThesis": "記事そのものより、AIが再利用できる元データを持つことが重要になる。",
  "audience": ["ひとりメディア運営者"],
  "audiencePain": "同じアイデアを媒体ごとに何度も書き直している。",
  "claims": [
    {
      "claim": "完成記事より構造化された元レコードのほうが再利用しやすい。",
      "supportingEvidence": "主張、根拠、具体例、反論を分けてAIに渡せるため。"
    }
  ],
  "platformAngles": [
    {
      "platform": "note",
      "hook": "ブログを記事の山ではなくDBとして見る。"
    },
    {
      "platform": "Threads",
      "hook": "ブログはAIが使う素材置き場になる。"
    }
  ]
}
```

これだけでも、普通の記事メモよりはAIに渡しやすくなります。

大事なのは、最初から完璧なスキーマを作ることではありません。AIが迷わず使えるように、考えの部品を分けて保存することです。

### ひとつの知識レコードから、複数の出力を作る

同じアイデアでも、媒体によって見せ方は変わります。

noteでは、背景から主張までをじっくり読ませる論考になります。

Substackでは、最近の気づきや制作過程を交えた手紙に近くなります。

Threadsでは、1投稿につき1メッセージの短い連投になります。

YouTubeでは、章立てされた解説台本になります。

でも、これらの中心にあるものは同じです。主張、読者、根拠、具体例は共通しています。

だからこそ、元になる知識レコードを整える価値があります。ひとつのレコードを作っておけば、媒体ごとにゼロから考え直すのではなく、同じ素材を別の形へ展開できます。

これは「AIで楽して量産する」という話ではありません。

AIが使える素材を、人間がちゃんと整えるという話です。

### 最初からAPI化しないほうがいい

ここで焦って、いきなりOpenAI APIやAnthropic APIを組み込む必要はありません。

むしろMVPでは、Codex、Claude Code、ChatGPT、Claudeアプリに、JSONレコードとプロンプトを手動で渡すくらいで十分です。

理由はシンプルです。

まだ分かっていないからです。

どのフィールドが出力品質に効くのか。どの媒体で追加情報が必要になるのか。どこで人間の編集が必要になるのか。どのプロンプトが弱いのか。

それらが分からないまま自動化すると、使えない自動化を作る可能性があります。

まずは手動で回して、何が効くかを見る。そのあとで、必要な部分だけ自動化する。

この順番のほうが、ひとりメディア運営者には合っています。

## 反論への回答

「ブログをDB化すると、創作が重くならないか」という不安は自然です。

すべてを最初から細かく構造化する必要はありません。

まずは、主張、読者、根拠、具体例、反論、トーンだけで十分です。

ここで、ObsidianとSanityの役割を分けて考えると分かりやすくなります。

Obsidianは、考える場所です。

まだ整理されていないメモ、リサーチ、開発ログ、ふと思いついた言葉、途中の違和感を置いておく場所です。

Sanityは、展開する場所です。

Obsidianの中から「これは何度も使う」と分かった知識を、主張、根拠、具体例、反論、媒体別の切り口として構造化して保存します。

そう考えると、構造化は創作を縛るものではなく、あとから再利用しやすくするための整理です。

## 締め

これからのブログは、単に記事を並べる場所ではなくなっていくと思います。

少なくとも、AIを発信ワークフローに入れるなら、完成記事より前の「元になる素材」をどう設計するかが重要になります。

記事を書く前に、まずひとつの知識レコードを作る。

そこに、主張、根拠、具体例、反論、読者、媒体別の切り口を入れる。

そのレコードから、note、Substack、Threads、YouTube、Podcast、図解、GitHubドキュメントへ展開する。

これが、AI時代のブログ運営の土台になるはずです。

## 人間による確認ポイント

- `coreThesis` が変わっていないか
- 元レコードにない主張を足していないか
- note記事として読める自然な流れになっているか
- 公開前に確認すべき根拠や表現はあるか
- 保存先は `outputs/note/` でよいか
